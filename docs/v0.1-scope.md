# LiveBoard v0.1 — First Release Scope

## Goal

A user can manage Kanban boards from the CLI, backed by Markdown files with automatic Git commits. No server, no UI — just the core read/write pipeline working end-to-end.

---

## In Scope

### Data Models (`pkg/models/`)

- `Board` — name, description, tags, columns
- `Column` — name, position, cards
- `Card` — id (uuidv7), title, completed, tags, assignee, priority, due date

### Markdown Parser (`internal/parser/`)

- Parse board `.md` file into `Board` model using goldmark
- Extract YAML frontmatter (name, description, tags)
- H2 headings → columns
- List items → cards (checkbox state, title)
- HTML comment → card ID (`<!-- liveboard:id=... -->`)
- Inline metadata lines → tags, assignee, priority, due

### Markdown Writer (`internal/writer/`)

- Apply mutations back to Markdown via AST transform
- Preserve formatting, comments, and custom content
- Produce minimal, clean Git diffs
- Never rewrite the entire file — transform only affected nodes

### Board Engine (`internal/board/`)

- Card CRUD: create, read, update, delete
- Card operations: move (across columns), complete/uncomplete, tag/untag
- Column CRUD: create, rename, reorder, delete
- ID assignment: auto-generate uuidv7 for new cards

### Workspace (`internal/workspace/`)

- Scan a directory for `.md` board files
- Load board by name
- Create new board with default columns (Backlog, In Progress, Review, Done)
- Delete board (remove file)
- Respect `.liveboard/config.yaml` for default columns

### Git Integration (`internal/git/`)

- Auto-commit after every write operation
- Structured commit messages: `card: add "title" → Column`
- Configurable: enable/disable auto-commit
- Initialize repo if workspace isn't already a Git repo

### CLI (`cmd/liveboard/`)

Using cobra:

```
liveboard board list
liveboard board create <name>
liveboard board delete <name>

liveboard card add <board> "<title>" [--column <col>]
liveboard card move <id> "<column>"
liveboard card complete <id>
liveboard card tag <id> <tag> [tag...]
liveboard card show <id>

liveboard column add <board> "<name>"
liveboard column move <board> "<name>" --after "<other>"
liveboard column delete <board> "<name>"
```

### Configuration

- Global config: `~/.config/liveboard/config.yaml`
- Project config: `.liveboard/config.yaml`
- Settings: default workspace path, default columns, git auto-commit toggle

---

## Out of Scope (deferred to v0.2+)

| Feature | Reason to defer |
|---------|----------------|
| REST API server | CLI proves the engine first; API is an HTTP wrapper on top |
| Web UI (LiveView) | Requires API server |
| Search index (bleve) | Not needed until board count grows |
| AI agent layer | Requires API server for tool calls |
| Cross-board card linking | Advanced feature; single-board usage comes first |
| Event bus / SSE / WebSocket | Only needed for real-time UI |
| File watcher (fsnotify) | Only needed for live-reload in UI |
| Card body/notes | Metadata-only cards are sufficient for v0.1 |
| Branch-per-board Git | Keep Git simple in v0.1 |

---

## Key Technical Risks

1. **Markdown round-tripping** — The parser/writer must preserve user formatting. This is the hardest problem and should be tackled first with thorough test cases.
2. **Card ID stability** — HTML comment IDs must survive edits by external editors (VS Code, Obsidian). Edge cases: user accidentally deletes the comment, duplicate IDs.
3. **Goldmark AST manipulation** — goldmark is primarily a renderer; AST mutation support may be limited. May need to operate on raw text with parser guidance.

---

## Implementation Order

1. `pkg/models` — Define types
2. `internal/parser` — Markdown → model (+ comprehensive tests)
3. `internal/writer` — Model → Markdown (+ round-trip tests)
4. `internal/board` — CRUD engine wiring parser + writer
5. `internal/workspace` — Folder scanning and board management
6. `internal/git` — Auto-commit integration
7. `cmd/liveboard` — CLI commands via cobra
8. Integration tests — End-to-end CLI → Markdown → Git flow

---

## Success Criteria

- `liveboard board create myboard` creates a valid Markdown file
- `liveboard card add myboard "Fix bug"` appends a card with a uuidv7 ID
- `liveboard card move <id> "Done"` moves the card and the Markdown diff is minimal
- `git log` shows a structured commit for every operation
- Editing the `.md` file externally and running CLI commands doesn't corrupt the board
